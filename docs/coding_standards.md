# Coding Standards and Guidelines

This document outlines the code quality, formatting, linting, and design patterns required for the Cognitive Voice Intelligence Platform. Adherence to these standards is enforced in CI/CD pipeline checks.

---

## 🐍 Python Coding Standards (Backend & Database)

### Style & Formatting
1.  **Code Style**: Strictly adhere to **PEP 8**.
2.  **Formatter**: Use **Black** for auto-formatting (line length = 88).
3.  **Linter**: Use **Ruff** for static analysis and import sorting.
4.  **Type Checking**: Enforce strict type hints. Run **Mypy** checks prior to commits:
    ```bash
    mypy backend/app
    ```

### Imports Conventions
*   Group imports in the following order:
    1.  Standard library imports.
    2.  Third-party packages (e.g., FastAPI, SQLAlchemy).
    3.  Local module imports (using absolute paths starting with `app.`).
*   Example:
    ```python
    import os
    from uuid import UUID

    from fastapi import APIRouter, Depends
    from sqlalchemy.ext.asyncio import AsyncSession

    from app.core.config import settings
    from app.database import get_db
    ```

## ⚡ TypeScript & React Coding Standards (Frontend)

### Style & Formatting
1.  **Formatting**: Enforce **Prettier** code formatting.
2.  **Linting**: Use **ESLint** with Next.js strict configurations.
3.  **Strict Typing**: Disable `any` types. All variables, function parameters, and API responses must have explicit interfaces or types.
4.  **Directory Structure**: Use alias imports prefixed with `@/` matching the Next.js standard.
    ```typescript
    import { RecordingWizard } from "@/components/RecordingWizard";
    import { SessionResponse } from "@/types/api";
    ```

---

## 🏗️ Monorepo Boundary Rules

1.  **Strict Isolation**: The frontend applications and the backend API code must remain completely decoupled.
    *   *Rule*: Never import Python backend code directly in TypeScript code or vice versa.
    *   *Rule*: Shared definitions must be synchronized via TypeScript types autogenerated from backend Pydantic schemas using code-generation scripts where applicable.
2.  **Database Separation**: The database directory acts as the data-source of truth.
    *   *Rule*: The FastAPI app must only access the database via dependency injection (`db: AsyncSession = Depends(get_db)`). Do not instantiate database engines inside endpoints or route handlers.

---

## 🛡️ Error Handling Guidelines

### FastAPI Core Boundaries
All endpoints must be protected against unexpected internal exceptions. We enforce a global exception handler in `backend/app/main.py`.

1.  **Pydantic Validation Errors**: Handled automatically by FastAPI, returning `422 Unprocessable Entity` with details.
2.  **Domain Exceptions**: Define custom subclasses extending standard HTTP exception modules.
    ```python
    class PlatformException(Exception):
        """Base exception for all system errors."""
        def __init__(self, message: str, status_code: int = 400):
            self.message = message
            self.status_code = status_code
    ```
3.  **ASR Processing Safeguards**: Wrap Whisper transcription execution in standard exception handlers. If transcription fails, set the database status to `failed` and log the traceback. Do not allow the FastAPI thread block to crash.

## 📝 Logging Guidelines

*   **Libraries**: Use the standard Python `logging` module. Avoid using raw `print()` statements.
*   **Format**: Use structured JSON log formats in staging/production to simplify analysis by log aggregate tools (e.g., Datadog, ELK).
*   **Log Levels**:
    *   `DEBUG`: System metrics calculations, variable arrays, file system write traces.
    *   `INFO`: Client request endpoints triggered, state changes (e.g., session initialized, processing finished).
    *   `WARNING`: Missing optional config variables, recovery from minor network hiccups.
    *   `ERROR`: Database read failures, audio corruption exceptions, Faster-Whisper decoding errors.
